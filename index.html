Scaling AI Applications: Interview Q&A

Sr. Manager

How would you scale this API to handle 10x more traffic?

Three-pronged approach:

  1. Response Caching (Redis) - Cache ingredient data (24-72h TTL), cache analysis reports by hash (1-6h TTL). Expected: 5x throughput.
  2. API Key Load Balancing - Pool multiple Gemini keys with rate-aware selection. N keys = N√ó capacity.
  3. Async Processing - Job queue (Celery/Redis Queue), return job ID immediately, poll for results.

Trade-off: Caching introduces stale data risk. Mitigation: Cache invalidation on safety updates, appropriate TTLs.

Director

Why did you choose Qdrant over other vector databases?

  • Free tier + cloud managed (1GB free)
  • Excellent hybrid search (vector + payload filtering)
  • Self-host option for cost optimization later
  • Simple REST API for debugging

vs Pinecone: More expensive, no self-host

Trade-off: Less mature than Pinecone. Mitigation: Active development, good docs.

Sr. Manager

How do you handle API rate limits from Gemini?

Scaled approach: Rate-limited key pool with multiple API keys. Track requests per key in sliding window. Rotate to available key or queue if all exhausted.

Trade-off: Multiple keys = more cost & complexity. Consider: Is traffic worth operational overhead?

Executive

Why multi-agent architecture instead of single LLM call?

Multi-agent (Research ‚Üí Analysis ‚Üí Critic):

  • Separation of concerns
  • Self-correction loop (Critic validates)
  • Better accuracy through validation gates
  • Easier to debug individual agents

Trade-off: 3x LLM calls = higher latency. For safety-critical info, accuracy > speed. Critic catches ~15% of issues.

Sr. Manager

How do you ensure consistency between mobile and web clients?

  1. Single REST API - Both clients call same /analyze endpoint
  2. Shared response schema - Pydantic models define contract
  3. API versioning - /api/v1/analyze for future changes

Trade-off: Both clients get same data even if one needs less. Accept slight over-fetching for consistency.

Director

How would you add real-time updates for long-running requests?

Options: Polling (simple, inefficient), WebSockets (complex, bidirectional), SSE (real-time, simple)

Recommendation: Server-Sent Events (SSE) - each stage emits progress event. Simple, works everywhere, one-way is sufficient.

Director

What's your testing strategy for LLM-based features?

  1. Unit tests - Mock LLM, test business logic
  2. Integration tests - Agent orchestration with fixtures
  3. Contract tests - Verify LLM output schema
  4. Evaluation tests - Accuracy on labeled datasets
  5. Load tests - Performance under stress

Key insight: LLM outputs are non-deterministic. Use temperature=0.1, test schema compliance not exact text.

Sr. Manager

How do you handle failures gracefully?

  • LLM timeout: Retry with exponential backoff
  • Rate limit (429): Switch to backup API key
  • Qdrant down: Fall back to Google Search only
  • Critic rejection: Retry up to 3x, then escalate
Executive

What would you do differently if starting over?

  1. Async from day one - Easier to add concurrency later
  2. Caching earlier - Would have saved dev API costs
  3. Structured outputs - Gemini JSON mode for reliable parsing
  4. Observability first - LangSmith from start
  5. Horizontal scaling design - Stateless API from beginning
Executive

How do you balance cost vs performance?

Cost per request: Gemini ~$0.01-0.05, Qdrant free tier, Railway ~$5/mo

Optimizations: Cache common ingredients (80% hit top 100), smaller models for validation, batch embeddings, appropriate TTLs.

Trade-off: Aggressive caching saves cost but risks stale safety data. Mitigation: 24h TTL + manual invalidation.
